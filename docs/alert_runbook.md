# Alert Runbook

This document provides response procedures for alerts generated by the trading system monitoring infrastructure.

## Alert Severity Levels

| Level | Description | Response Time | Notification |
|-------|-------------|---------------|--------------|
| **CRITICAL** | System-impacting issues requiring immediate attention | < 5 minutes | Slack + Email |
| **WARNING** | Potential issues that need investigation | < 30 minutes | Slack |
| **INFO** | Informational alerts, no action required | N/A | Slack |

---

## Critical Alerts

### Daily Loss Limit Exceeded

**Alert Name:** `daily_loss_limit`

**Description:** The system has reached or exceeded the configured daily loss limit. Trading has been automatically halted.

**Symptoms:**
- `trading_halted` gauge = 1
- `daily_pnl_dollars` below configured limit

**Response:**
1. Verify the alert is not a false positive by checking:
   - Recent trade history
   - Current positions
   - Market conditions
2. Review the trades that led to the loss
3. Assess if any positions need manual intervention
4. Do NOT re-enable trading until:
   - Root cause is understood
   - Risk parameters are reviewed
   - Team lead approval is obtained

**Recovery:**
```python
# After approval, reset daily P&L tracking
risk_manager.reset_daily_pnl()
risk_manager.enable_trading()
```

---

### Position Limit Exceeded

**Alert Name:** `position_limit`

**Description:** A position has exceeded the configured size limit.

**Context Fields:**
- `ticker`: The ticker with the oversized position
- `current_size`: Current position size
- `limit`: Configured limit

**Response:**
1. Check if this is due to:
   - Partial fills accumulating
   - Failed cancellation orders
   - Configuration error
2. Reduce position to within limits if needed
3. Verify order management system is functioning

**Recovery:**
```python
# Close excess position
order_manager.reduce_position(ticker, target_size=limit)
```

---

### WebSocket Disconnected

**Alert Name:** `websocket_disconnect`

**Description:** WebSocket connection to a trading platform has been lost.

**Context Fields:**
- `platform`: Affected platform
- `downtime`: Duration of disconnection

**Response:**
1. Check platform status page for outages
2. Verify network connectivity
3. Check API credentials are still valid
4. Monitor for automatic reconnection

**If no auto-reconnect after 2 minutes:**
```bash
# Restart WebSocket connection
python -m src.core.websocket_client --platform kalshi --reconnect
```

---

### API Error

**Alert Name:** `api_error`

**Description:** Repeated API errors from a trading platform.

**Context Fields:**
- `platform`: Affected platform
- `endpoint`: Failing endpoint
- `error`: Error message

**Response:**
1. Check error type:
   - **401/403**: Authentication issue - rotate API keys
   - **429**: Rate limiting - reduce request frequency
   - **500+**: Platform issue - check status page
2. Review recent changes to API client
3. Verify API key permissions

## Warning Alerts

### Risk Utilization High

**Alert Name:** `risk_breach`

**Description:** Risk utilization has exceeded warning threshold (80%).

**Context Fields:**
- `limit_type`: Type of limit approaching
- `utilization_pct`: Current utilization percentage

**Response:**
1. Review current positions
2. Consider reducing exposure proactively
3. Verify no stuck orders are inflating position counts

---

### Execution Failure

**Alert Name:** `execution_failure`

**Description:** A trade execution has failed.

**Context Fields:**
- `order_id`: Failed order ID
- `ticker`: Ticker symbol
- `reason`: Failure reason

**Response:**
1. Check order status on platform directly
2. Verify market is still tradeable
3. Review order parameters (size, price, side)
4. Retry order if appropriate

---

## System Health Alerts

### Component Unhealthy

**Alert Name:** `system_health`

**Description:** A system component health check has failed.

**Context Fields:**
- `component`: Failed component
- `status`: Current status
- `details`: Additional details

**Response:**
1. Check component logs for errors
2. Verify dependencies (database, external APIs)
3. Restart component if needed

**Component-specific responses:**

| Component | Check | Recovery |
|-----------|-------|----------|
| database | `SELECT 1` works | Restart database connection pool |
| api_kalshi | API responds | Verify credentials, check rate limits |
| api_polymarket | API responds | Verify wallet connection |
| websocket_* | Connection active | Trigger reconnect |

## Metrics Reference

### Key Metrics to Monitor

| Metric | Description | Warning Threshold | Critical Threshold |
|--------|-------------|-------------------|-------------------|
| `daily_pnl_dollars` | Daily P&L | < -$100 | < -$200 |
| `risk_utilization_pct{limit_type="daily_loss"}` | Loss limit usage | > 80% | > 95% |
| `errors_total` rate | Error rate | > 1/min | > 5/min |
| `api_request_latency_seconds` p99 | API latency | > 2s | > 5s |
| `websocket_connected` | WS status | 0 for > 30s | 0 for > 2min |

### Prometheus Queries

**Error rate by component:**
```promql
rate(errors_total[5m]) by (component, error_type)
```

**API latency percentiles:**
```promql
histogram_quantile(0.99, rate(api_request_latency_seconds_bucket[5m]))
```

**Trading activity:**
```promql
rate(arbitrage_trades_executed_total[5m]) by (platform, status)
```

## Post-Incident

After resolving any CRITICAL alert:

1. **Document the incident:**
   - Timeline of events
   - Root cause
   - Resolution steps
   - Impact (P&L, missed opportunities)

2. **Identify improvements:**
   - Could we detect this sooner?
   - Should thresholds be adjusted?
   - Are there missing alerts?

3. **Update runbook** if response procedures were unclear or incorrect
